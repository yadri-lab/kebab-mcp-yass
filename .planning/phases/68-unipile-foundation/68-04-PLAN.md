    ```typescript
    import { createHash, randomUUID } from "node:crypto";
    import { getContextKVStore } from "@/core/request-context";

    /** D-08: audit log TTL is 90 days. Upstash respects EX; FilesystemKV ignores (dev only). */
    export const AUDIT_TTL_SECONDS = 90 * 24 * 60 * 60; // 7,776,000

    /** D-05: result enum (locked). NEVER add 'pending'. */
    export type AuditResult =
      | "success"
      | "unverified_timeout"
      | "error_rate_limit"
      | "error_account_restricted"
      | "error_not_connected"
      | "error_unipile_5xx";

    /** D-07: audit row schema. note_text is NEVER persisted — only params_hash. */
    export interface AuditRow {
      audit_id: string;
      actor_user_id: string;
      tool: string;
      account_id: string;
      params_hash: string;
      result: AuditResult;
      verified: boolean;
      dedup_hit: boolean;
      timestamp: string; // ISO-8601 UTC
    }

    /** Generate a UUIDv4 audit_id. */
    export function generateAuditId(): string {
      return randomUUID();
    }

    /**
     * D-05: SHA-256 over {tool, profile_url_normalized, note}, truncated to 16 hex chars.
     * Deterministic: keys are sorted before stringify so insertion order does not matter.
     * Caller MUST pass the already-normalized profile URL.
     */
    export function computeParamsHash(input: {
      tool: string;
      profile_url_normalized: string;
      note: string; // "" if absent — empty string is canonical
    }): string {
      // Canonical form: sort keys, then JSON.stringify
      const canonical = JSON.stringify({
        note: input.note,
        profile_url_normalized: input.profile_url_normalized,
        tool: input.tool,
      });
      return createHash("sha256").update(canonical).digest("hex").slice(0, 16);
    }

    /**
     * Write the audit row + hash pointer index. Both expire after 90 days (D-08).
     * Tenant-scoped via getContextKVStore (D-18).
     */
    export async function writeAuditRow(row: AuditRow): Promise<void> {
      const kv = getContextKVStore();
      const rowJson = JSON.stringify(row);
      await Promise.all([
        kv.set(`unipile:audit:${row.audit_id}`, rowJson, AUDIT_TTL_SECONDS),
        kv.set(`unipile:audit:hash:${row.params_hash}`, rowJson, AUDIT_TTL_SECONDS),
      ]);
    }

    /**
     * Look up a prior audit row by params_hash. Returns the row if present, null otherwise.
     * Used by tool handlers to enforce dedup (D-05/D-06: caller CANNOT bypass).
     */
    export async function checkDedup(paramsHash: string): Promise<AuditRow | null> {
      const kv = getContextKVStore();
      const raw = await kv.get(`unipile:audit:hash:${paramsHash}`);
      if (!raw) return null;
      try {
        const parsed = JSON.parse(raw) as AuditRow;
        if (parsed && typeof parsed.audit_id === "string") return parsed;
        return null;
      } catch {
        return null;
      }
    }
    ```

    Critical design notes:
    - The hash pointer stores the FULL row (JSON.stringify(row)) rather than just the audit_id pointer. Reason: one KV read serves the entire dedup-check + display-of-prior-result need. Trade-off: 2x storage cost for the row body. At Cadens scale (5-30/day × 90 days = ~2700 rows max), this is negligible. Document this in the SUMMARY.
    - Canonical JSON: keys are sorted alphabetically in the literal — `{note, profile_url_normalized, tool}`. This guarantees the SAME hash regardless of how the caller orders the input object's keys.
    - There is NO `dedup_key` or `bypass_dedup` parameter anywhere (D-06).
    - There is NO function that writes `note` to KV (D-07).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tool handler → KV (audit log writes) | Tenant-scoped via getContextKVStore — tenant A cannot read tenant B's audit history. |
| LLM caller → audit module | LLM CANNOT bypass dedup (D-06): no `bypass_dedup` / `dedup_key` param exists in any exported function signature. Verified by export-shape test. |
| KV row → operator dashboard (phase 71) | Audit rows contain `actor_user_id`, `tool`, `account_id`, `params_hash` — no PII in the value. The hash is one-way; reverse lookup requires the original note. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-68-04-01 | Information Disclosure | Audit row contains LinkedIn note PII | mitigate | D-07: `note_text` is NEVER persisted. Only `params_hash` (SHA-256, 16 hex chars). Verified by test "row JSON does NOT contain a 'note' or 'note_text' field". |
| T-68-04-02 | Tampering | LLM bypasses dedup by varying note text by 1 char | mitigate | D-05 by design: 1-char change = new hash = new call allowed. This is the explicit feature — protects against re-spam with the SAME note while preserving legitimate re-engagement with a different message. |
| T-68-04-03 | Tampering | LLM passes `dedup_key: 'force'` to bypass dedup | mitigate | D-06: no such parameter exists in the function signatures. Tests verify the export shape. TypeScript at compile time + runtime test = double layer. |
| T-68-04-04 | Repudiation | Audit row missing actor_user_id | mitigate | `actor_user_id` is required in the `AuditRow` interface (TypeScript-enforced). Caller MUST supply it; the tool handler in Plan 06 will accept and pass through `actor_user_id` from the tool args. |
| T-68-04-05 | Denial of Service | Audit log grows unbounded | mitigate | D-08: 90-day TTL on Upstash. At Cadens scale (max 30/day × 90 = 2700 rows × ~300 bytes each = ~800 KB) — trivial. |
| T-68-04-06 | Information Disclosure | Hash collision exposes prior row to attacker who crafts the same (tool, url, note) combination | accept | SHA-256 truncated to 16 hex chars = 64 bits. Birthday-bound at ~4.3 billion entries — far above any realistic audit volume. Collision risk negligible.
