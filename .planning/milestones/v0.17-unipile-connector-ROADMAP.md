# Milestone v0.17 — Unipile Connector

**Started:** 2026-05-18
**Goal:** Ship a dedicated `src/connectors/unipile/` connector exposing 9 high-level write tools for LinkedIn (5) and WhatsApp (4), built on the Unipile Node SDK. Replaces the failed Browserbase-based LinkedIn write attempt documented in [ADR 0001](../../docs/adr/0001-unipile-as-linkedin-whatsapp-write-provider.md). Browserbase + Apify connectors remain in scope for LinkedIn read-only operations.

## Decision Source

- **[ADR 0001](../../docs/adr/0001-unipile-as-linkedin-whatsapp-write-provider.md)** — Unipile chosen over LinkedAPI, Browserbase Scale, and self-hosted Playwright
- **2026-05-18 incident** — Browserbase `web_act` failed silently on a real LinkedIn connect attempt (Antoine Vercken / Trusk)
- **GPT review feedback** (2026-05-18 conversation transcript) — adopted 4 architectural points: dedicated webhook route, durable KV (no in-memory queues on Vercel), fail-closed rate limiting, outbox pattern for CRM sync

## Phase Overview

| # | Phase | Goal | Requirements | Effort |
|---|-------|------|--------------|--------|
| 68 | Unipile foundation | Client + manifest + first write E2E (send_connection + get_relationship_status) + CRM outbox + audit log | UNI-01..UNI-06 | 2.5 days |

| 70 | Webhooks + WhatsApp V1 | `/api/unipile/webhook` (signed, idempotent), `account.status` + `new_relation` + `new_message` handlers, 4 WhatsApp tools (send_message, list_chats, get_conversation, list_contacts) | UNI-12..UNI-19 | 3 days |
| 71 | Hardening | Kill switches, fail-closed rate limit, metrics, audit query API, docs, multi-tenant verification | UNI-20..UNI-24 | 1.5 days |

**Total:** ~9 days focused. 4 phases. 9 tools shipped (5 LinkedIn + 4 WhatsApp).

- [ ] **UNI-04:** `src/connectors/unipile/lib/audit.ts` — writes `{actor_user_id, tool, account_id, params_hash, result, verified, dedup_hit, timestamp, audit_id}` to KV under `audit:unipile:<audit_id>`. PII (message bodies) excluded. Returns `audit_id` for caller correlation.
- [ ] **UNI-05:** `src/connectors/unipile/lib/crm-bridge.ts` — abstract `CrmAdapter` interface. V1 Twenty adapter via custom webhook (calls `process.env.UNIPILE_CRM_WEBHOOK_URL` with HMAC signature). Outbox pattern: write `{audit_id, crm_log, status: 'pending'}` to KV, dispatch async, status transitions `pending → synced | failed`. **No claim of atomicity** between Unipile and CRM (documented).
- [ ] **UNI-06:** `linkedin_send_connection` tool — params `{profile_url, note?, allow_existing_pending?, crm_log?, dedup_key?}`, returns standard result envelope `{provider_ok, verified, crm_sync, dedup_hit, audit_id, next_check_at?, invitation_id?}`. Verify-after-write with 3-poll backoff (2s/5s/10s). On timeout returns `verified: 'pending'` (NOT `'false'`) and sets `next_check_at`. `linkedin_get_relationship_status` tool — params `{profile_url}`, returns `{degree: 1|2|3|null, connection_status: 'none'|'pending'|'connected', last_message_at?, has_replied?}`. Integration test against Unipile sandbox or mocked SDK.

### Phase 69 — LinkedIn Writes Completion (UNI-B) — PENDING

- [ ] **UNI-07:** `linkedin_send_message` — 1st-degree DM only. Returns `error: 'not_connected'` if 2nd/3rd degree. Accepts `attachments?` (PDF/image, ≤15MB). Verify-after-write via webhook `message.sent` (10s wait) → `verified: 'pending'` if no webhook receipt.
- [ ] **UNI-08:** `linkedin_send_inmail` — explicit tool, requires `allow_inmail: true` and optional `max_inmail_credits` cap. Returns `{credits_used, credits_remaining}` from Unipile response. Rejects if account lacks Premium/Sales Nav.
- [ ] **UNI-09:** `linkedin_engage` super-tool — params `{profile_url, message?, fallback_if_unreachable?: 'skip'|'inmail', allow_inmail?, dry_run?, crm_log?, dedup_key?}`. Routes to: (a) `send_message` if 1st-degree, (b) `send_connection` if 2nd/3rd reachable, (c) `send_inmail` if out-of-network AND `fallback='inmail'` AND `allow_inmail`, (d) skip with reason otherwise. Returns discriminated union typed result. `dry_run: true` returns proposed action + risks **without executing** any provider call. Default `dry_run: false`.
- [ ] **UNI-10:** `linkedin_list_pending` — `{account_id?, older_than_days?}`, returns array of invitations with `{invitation_id, recipient_profile_url, recipient_name, sent_at, age_days, has_note}`. Useful for cleanup loops.
- [ ] **UNI-11:** Rate-limiter module `src/connectors/unipile/lib/rate-limiter.ts` — per-account daily/weekly counters in KV, `LINKEDIN_DAILY_CONNECT_CAP` (default 25), `LINKEDIN_WEEKLY_CONNECT_CAP` (100), `LINKEDIN_DAILY_DM_CAP` (50), `LINKEDIN_DAILY_INMAIL_CAP` (15). **Fail-closed by default** (no KV = block) with `KEBAB_UNIPILE_RATELIMIT_FAIL_MODE=open` escape hatch. Returns `{blocked: true, retry_after, daily_used, daily_limit}` not exception.

### Phase 70 — Webhooks + WhatsApp V1 (UNI-C) — PENDING

- [ ] **UNI-12:** `app/api/unipile/webhook/route.ts` (dedicated, NOT the generic webhook router) — POST handler verifies `Unipile-Auth` HMAC header (per-tenant secret), idempotency via `event_id` dedup key in KV (24h TTL), dispatches to event handlers, replies 200 in <30s. Heavy work (CRM propagation, notifications) async via background task or queued.
- [ ] **UNI-13:** Handler `account.status` — when status transitions to error states (`credentials_expired`, `restricted`), sets connector-level halt flag in KV per `account_id`, surfaces in dashboard `/config → Connectors`, optionally notifies owner via existing notification channel.
- [ ] **UNI-14:** Handler `new_relation` (LinkedIn connect accepted) — updates linked CRM via outbox (transition status, append note), optionally triggers next-step prompt in CLI/UI.
- [ ] **UNI-15:** Handler `new_message` (LI + WA inbound) — updates CRM `last_replied_at`, content hash only (no PII), optional Slack notification opt-in.
- [ ] **UNI-16:** `whatsapp_send_message` — params `{to, text?, attachments?, reply_to_message_id?, crm_log?, dedup_key?}`. `to` accepts E.164 phone, contact name (resolved via list_contacts), or existing `chat_id`. Attachments support image/doc/audio/video ≤15MB. Standard result envelope.
- [ ] **UNI-17:** `whatsapp_list_chats` — `{account_id?, unread_only?, since?, limit?}`. Returns `[{chat_id, attendee_name, last_message_preview, last_message_at, unread_count}]`. Default `limit=20`, max `100`.
- [ ] **UNI-18:** `whatsapp_get_conversation` — `{chat_id, before?, limit?}`. Returns paginated messages. Default `limit=50`, max `200`. Includes attachments metadata (urls expire per Unipile policy — documented).
- [ ] **UNI-19:** `whatsapp_list_contacts` — `{account_id?, query?}`. Returns `[{contact_id, name, phone_e164, has_chat}]`. Used by resolver in `send_message`.

### Backlog — discovered during live test 2026-05-18 (route to phase 69 or 71)

- [ ] **UNI-25:** URL normalization — strip query strings before slug match. Currently `linkedin.com/in/foo?originalSubdomain=fr` (the format LinkedIn search results use natively) fails the `SLUG_RE` regex with terminal `\/?$`, surfacing as `error_unipile_5xx` to the caller instead of the more accurate `error_invalid_url`. Two fixes possible: (a) extend `SLUG_RE` to allow optional `\?[^/]*`, or (b) early-pre-process input via `new URL(input).pathname` before regex. Add tests for the 3 LinkedIn URL flavors with query strings (`?originalSubdomain=fr`, `?miniProfileUrn=...`, `?utm_source=...`).
- [ ] **UNI-26:** Error taxonomy refinement — Unipile 422 (`errors/invalid_recipient` "Recipient cannot be reached", `errors/invalid_parameters`) and 400 (`errors/invalid_parameters`) currently classify as `error_unipile_5xx` (the catch-all bucket). Inspect `classifyUnipileError()` and add explicit branches: 422 invalid_recipient → `error_recipient_unreachable` (new enum member), 400 → `error_invalid_request` (new enum member). Verified live 2026-05-18: send-self on Yassine's own profile returned `error_unipile_5xx` when the upstream actual error was a 422 from Unipile (LinkedIn rejects self-connect attempts).

## Success Criteria

- All 4 phases complete with green CI (lint, typecheck, test:unit, test:registry, test:contract, size:check)
- 9 tools registered, exposed via MCP, visible in `/config → Tools` sandbox
- Real prospect connect via `linkedin_engage` works end-to-end with Twenty CRM update (re-test Antoine Vercken flow that failed 2026-05-18)
- WhatsApp send + list + read works end-to-end on a personal WA account
- Webhook ingress proven by triggering a real `new_relation` event (accept own connection request from a 2nd account)
- Kill switches verified by toggling env var and confirming tools refuse
- Zero new bundle bytes regression > 10% headroom; new connector contributes <50KB compressed
- Multi-tenant smoke test passes for 2 accounts under one Unipile API key

## Out of scope (deferred to v0.18+)

- **ARCH-A from v0.16** (split vault/skills/api stores) — orthogonal to Unipile; revisit when characterization tests are written.
- **CRM adapters beyond Twenty custom webhook** (HubSpot, Pipedrive native) — only build when a real second tenant demands it.
- **`linkedin_search_people` / `linkedin_warm_prospect` / outreach sequences** (V2 in product brief) — explicit P5+ scope, conditional on V1 traction.
- **WhatsApp groups creation, reactions, read receipts management** — V1 covers send + read; group writes are V2.
- **Unipile Email + Calendar channels** — out of scope for this milestone (Google Workspace connector already covers these for Kebab's primary use case).
